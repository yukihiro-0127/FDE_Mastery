ベクトルデータベースとは

📚 ベクトルデータベースの定義

ベクトルデータベース（Vector Database）とは、データを数値ベクトル（多次元配列）として保存し、ベクトル間の類似度を高速に計算・検索できるデータベースです。

従来のDB: キーワード完全一致で検索（「りんご」で検索すると「りんご」だけヒット）

ベクトルDB: 意味的類似性で検索（「りんご」で検索すると「果物」「赤い」「甘い」もヒット）

💡 なぜベクトルDBが必要なのか

問題: 従来のデータベースは「完全一致」しか検索できない

  • 「iPhone 15」で検索 → 「iPhone15」（スペースなし）はヒットしない
  • 「顧客満足度向上」で検索 → 「CS改善」はヒットしない（同じ意味なのに）
  • 「エラーが出た」で検索 → 「問題が発生した」はヒットしない

解決策: ベクトルDBは「意味」で検索できる

  • 「iPhone 15」で検索 → 「iPhone15」「アイフォン15」「最新iPhone」すべてヒット
  • 「顧客満足度向上」で検索 → 「CS改善」「ユーザー体験向上」「NPS向上」もヒット
  • 「エラーが出た」で検索 → 「問題が発生」「不具合」「バグ」もヒット

ベクトル化（Embedding）の仕組み

🔢 ベクトル化とは

ベクトル化（Embedding）: テキストや画像を数値の配列に変換すること

テキスト: "りんご" → ベクトル: [0.2, 0.8, 0.1, 0.5, ...]（1536次元）
テキスト: "果物" → ベクトル: [0.3, 0.7, 0.2, 0.4, ...]（1536次元）

重要: 意味が似ているテキストは、ベクトルも似た値になる

1. テキストのベクトル化

入力: 「AIは人間の仕事を奪うのか」

出力: [0.12, -0.34, 0.56, ..., 0.78]（1536次元）

使用モデル: OpenAI text-embedding-3-small

2. 画像のベクトル化

入力: 猫の画像

出力: [0.45, 0.23, -0.12, ..., 0.89]（512次元）

使用モデル: CLIP、ResNet

3. 音声のベクトル化

入力: 「こんにちは」の音声

出力: [0.67, -0.45, 0.23, ..., 0.12]（768次元）

使用モデル: Whisper、Wav2Vec

4. 動画のベクトル化

入力: 商品紹介動画

出力: [0.34, 0.56, -0.78, ..., 0.23]（2048次元）

使用モデル: VideoMAE、TimeSformer

⚠️ ベクトル化の注意点

  • 次元数: 高次元ほど精度が高いが、計算コストも高い（通常512〜1536次元）
  • モデル選択: 日本語に強いモデルを選ぶ（OpenAI、Cohere、日本語特化モデル）
  • 正規化: ベクトルの長さを1に正規化することで、計算が高速化
  • バッチ処理: 大量のテキストを一度にベクトル化すると効率的

類似度計算の仕組み

📐 類似度計算の3つの方法

ベクトル間の「近さ」を計算する方法は主に3つあります。

1. コサイン類似度（Cosine Similarity）

定義: 2つのベクトルの角度を計算（-1〜1の値）

cos(θ) = (A · B) / (||A|| × ||B||)

例:
ベクトルA: [1, 2, 3]
ベクトルB: [2, 4, 6]
→ コサイン類似度 = 1.0（完全に同じ方向）

ベクトルA: [1, 0, 0]
ベクトルB: [0, 1, 0]
→ コサイン類似度 = 0.0（直交）

特徴: ベクトルの大きさに影響されない（方向だけを見る）

用途: テキスト類似度、推薦システム

2. ユークリッド距離（Euclidean Distance）

定義: 2つのベクトル間の直線距離

distance = √((A₁-B₁)² + (A₂-B₂)² + ... + (Aₙ-Bₙ)²)

例:
ベクトルA: [1, 2]
ベクトルB: [4, 6]
→ 距離 = √((1-4)² + (2-6)²) = √(9 + 16) = 5.0

特徴: 直感的にわかりやすい（物理的な距離）

用途: 画像検索、異常検知

3. ドット積（Dot Product）

定義: 2つのベクトルの要素ごとの積の合計

dot_product = A₁×B₁ + A₂×B₂ + ... + Aₙ×Bₙ

例:
ベクトルA: [1, 2, 3]
ベクトルB: [4, 5, 6]
→ ドット積 = 1×4 + 2×5 + 3×6 = 4 + 10 + 18 = 32

特徴: 計算が最も高速

用途: 大規模検索、リアルタイム推薦

計算方法 計算速度 精度 用途
コサイン類似度 中速 高 テキスト検索、推薦
ユークリッド距離 中速 中 画像検索、異常検知
ドット積 高速 中 大規模検索、リアルタイム

主要なベクトルデータベース

1. Pinecone ⭐⭐⭐⭐⭐

特徴: フルマネージド、最も使いやすい

料金: 無料枠あり（100万ベクトルまで）

用途: スタートアップ、プロトタイプ

メリット: セットアップ不要、自動スケーリング

デメリット: ベンダーロックイン、高コスト

2. Weaviate ⭐⭐⭐⭐

特徴: オープンソース、GraphQL対応

料金: 無料（セルフホスト）

用途: エンタープライズ、オンプレミス

メリット: 柔軟性が高い、コスト削減

デメリット: 運用が必要、学習コスト

3. Qdrant ⭐⭐⭐⭐

特徴: Rust製、高速

料金: 無料（セルフホスト）、クラウド版あり

用途: 高速検索、大規模データ

メリット: 高性能、フィルタリング機能

デメリット: 日本語ドキュメント少ない

4. Chroma ⭐⭐⭐

特徴: Python製、LangChain統合

料金: 無料（オープンソース）

用途: 開発環境、小規模プロジェクト

メリット: 簡単に始められる、軽量

デメリット: 本番環境には不向き

5. Milvus ⭐⭐⭐⭐

特徴: 大規模データ対応、分散処理

料金: 無料（オープンソース）

用途: 数億ベクトル以上の大規模検索

メリット: スケーラビリティ、高性能

デメリット: セットアップが複雑

6. pgvector（PostgreSQL拡張）⭐⭐⭐

特徴: 既存のPostgreSQLに追加

料金: 無料

用途: 既存システムへの追加

メリット: 学習コスト低い、既存DBと統合

デメリット: 大規模データには不向き

ベクトルDBの実装例

1. Pineconeを使った実装（Python）

import pinecone
from openai import OpenAI

# 1. Pineconeの初期化
pinecone.init(api_key="your-api-key", environment="us-west1-gcp")
index = pinecone.Index("my-index")

# 2. OpenAIでテキストをベクトル化
client = OpenAI(api_key="your-openai-key")
response = client.embeddings.create(
    model="text-embedding-3-small",
    input="AIは人間の仕事を奪うのか"
)
vector = response.data[0].embedding  # [0.12, -0.34, 0.56, ...]

# 3. ベクトルをPineconeに保存
index.upsert(vectors=[
    ("doc1", vector, {"text": "AIは人間の仕事を奪うのか"})
])

# 4. 類似検索
query_vector = client.embeddings.create(
    model="text-embedding-3-small",
    input="人工知能と雇用の関係"
).data[0].embedding

results = index.query(vector=query_vector, top_k=5, include_metadata=True)
for match in results.matches:
    print(f"類似度: {match.score}, テキスト: {match.metadata['text']}")

2. Chromaを使った実装（Python）

import chromadb
from chromadb.utils import embedding_functions

# 1. Chromaの初期化
client = chromadb.Client()
collection = client.create_collection(
    name="my_collection",
    embedding_function=embedding_functions.OpenAIEmbeddingFunction(
        api_key="your-openai-key",
        model_name="text-embedding-3-small"
    )
)

# 2. ドキュメントを追加（自動でベクトル化）
collection.add(
    documents=["AIは人間の仕事を奪うのか", "機械学習の基礎"],
    ids=["doc1", "doc2"]
)

# 3. 類似検索
results = collection.query(
    query_texts=["人工知能と雇用の関係"],
    n_results=5
)
print(results)

3. pgvectorを使った実装（SQL）

-- 1. pgvector拡張を有効化
CREATE EXTENSION vector;

-- 2. ベクトル列を持つテーブルを作成
CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    content TEXT,
    embedding VECTOR(1536)  -- 1536次元のベクトル
);

-- 3. ベクトルを挿入
INSERT INTO documents (content, embedding) VALUES
('AIは人間の仕事を奪うのか', '[0.12, -0.34, 0.56, ...]');

-- 4. 類似検索（コサイン類似度）
SELECT content, 1 - (embedding <=> '[0.15, -0.30, 0.60, ...]') AS similarity
FROM documents
ORDER BY embedding <=> '[0.15, -0.30, 0.60, ...]'
LIMIT 5;

ベクトルDBのユースケース

1. RAG（検索拡張生成）

用途: 社内文書検索、カスタマーサポート

仕組み: 質問をベクトル化 → 類似文書を検索 → LLMに渡して回答生成

例: 「退職金の計算方法は？」→ 就業規則から関連箇所を検索 → GPT-4で回答

2. 推薦システム

用途: 商品推薦、コンテンツ推薦

仕組み: ユーザーの行動をベクトル化 → 類似ユーザーを検索 → 類似ユーザーが好む商品を推薦

例: Netflix、Amazon、YouTube

3. 画像検索

用途: 類似画像検索、重複検出

仕組み: 画像をベクトル化 → 類似画像を検索

例: Google画像検索、Pinterest

4. 異常検知

用途: セキュリティ、品質管理

仕組み: 正常データをベクトル化 → 新データとの距離を計算 → 距離が大きい = 異常

例: クレジットカード不正検知、製造業の不良品検出

5. セマンティック検索

用途: 社内ナレッジベース、FAQ検索

仕組み: 質問と回答をベクトル化 → 意味的に類似した質問を検索

例: 「パスワードを忘れた」→「ログインできない」「認証エラー」もヒット

6. 重複検出

用途: スパム検出、コピーコンテンツ検出

仕組み: 文書をベクトル化 → 類似度が高い = 重複

例: ニュース記事の重複検出、SNSのスパム検出

ベクトルDBの選定基準

判断フローチャート

Step 1: データ規模を確認

〜100万ベクトル: Chroma、pgvector

理由: 軽量で簡単、既存DBに統合可能

100万〜1億ベクトル: Pinecone、Weaviate、Qdrant

理由: スケーラビリティとパフォーマンスのバランス

1億ベクトル以上: Milvus

理由: 分散処理、大規模データ対応

Step 2: 運用体制を確認

運用リソースなし: Pinecone（フルマネージド）

理由: セットアップ不要、自動スケーリング

運用リソースあり: Weaviate、Qdrant、Milvus

理由: コスト削減、カスタマイズ可能

Step 3: コスト予算を確認

低予算: Chroma、pgvector、Weaviate（セルフホスト）

理由: 無料、オープンソース

中予算: Pinecone（無料枠 + 有料プラン）

理由: 運用コスト削減、スケーラビリティ

高予算: Pinecone（エンタープライズプラン）

理由: SLA保証、専任サポート

ベクトルDBのパフォーマンス最適化

1. インデックス最適化

HNSW（Hierarchical Navigable Small World）: 最も高速なインデックス

IVF（Inverted File Index）: メモリ効率が良い

選択基準: 検索速度重視 → HNSW、メモリ重視 → IVF

2. 次元削減

PCA（主成分分析）: 1536次元 → 512次元に削減

効果: 検索速度2〜3倍、メモリ使用量1/3

トレードオフ: 精度が若干低下（通常5%以下）

3. バッチ処理

方法: 複数のベクトルを一度に検索

効果: スループット向上、コスト削減

例: 100件を1件ずつ検索 → 100回のAPI呼び出し、100件を一度に検索 → 1回のAPI呼び出し

4. キャッシング

方法: よく検索されるクエリの結果をキャッシュ

効果: レスポンス時間短縮、コスト削減

実装: Redis、Memcached

ベクトルDBの課題と対策

⚠️ 課題1: コールドスタート問題

問題: 新規ユーザーや新商品のベクトルがない

対策:

  • メタデータ（カテゴリ、タグ）を使ったハイブリッド検索
  • 人気商品を初期推薦
  • ユーザーにアンケートで好みを聞く

⚠️ 課題2: ベクトルの更新

問題: データが更新されたらベクトルも再計算が必要

対策:

  • 差分更新: 変更された部分だけ再計算
  • バッチ更新: 夜間に一括更新
  • リアルタイム更新: ストリーミング処理

⚠️ 課題3: 多言語対応

問題: 日本語と英語で別のベクトルになる

対策:

  • 多言語対応モデルを使用（mBERT、XLM-R）
  • 翻訳してから検索
  • 言語ごとに別のインデックスを作成
