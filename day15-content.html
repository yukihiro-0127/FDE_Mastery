Day 8-10: コード読解の完全復習

1. JSON（JavaScript Object Notation）

JSONの本質

JSONとは: データを「キー: 値」のペアで表現する形式

{
  "name": "田中太郎",
  "age": 30,
  "skills": ["Python", "SQL", "AI"],
  "address": {
    "city": "東京",
    "zip": "100-0001"
  }
}

読み方のコツ:

  • {}（波括弧）= オブジェクト（辞書）
  • []（角括弧）= 配列（リスト）
  • ネストを追いかける（階層構造を理解）

2. SQL（Structured Query Language）

SQLの本質

SQLとは: データベースに「何をしたいか」を伝える言語

SELECT name, age 
FROM users 
WHERE age >= 30 
ORDER BY age DESC 
LIMIT 10;

読み方のコツ:

  • SELECT = 何を取得するか
  • FROM = どのテーブルから
  • WHERE = どんな条件で
  • ORDER BY = どう並べるか
  • LIMIT = 何件取得するか

3. Python

Pythonの本質

Pythonとは: 人間が読みやすいプログラミング言語

def calculate_total(items):
    total = 0
    for item in items:
        total += item['price']
    return total

result = calculate_total([
    {'name': 'りんご', 'price': 100},
    {'name': 'みかん', 'price': 80}
])
print(result)  # 180

読み方のコツ:

  • def = 関数の定義
  • for = 繰り返し
  • if = 条件分岐
  • return = 結果を返す
  • インデント（字下げ）でブロックを表現

4. API（Application Programming Interface）

APIの本質

APIとは: プログラム同士が会話するための「窓口」

POST /api/users
Content-Type: application/json

{
  "name": "田中太郎",
  "email": "tanaka@example.com"
}

→ レスポンス:
{
  "id": 123,
  "name": "田中太郎",
  "created_at": "2024-01-01T10:00:00Z"
}

読み方のコツ:

  • GET = データを取得
  • POST = データを作成
  • PUT = データを更新
  • DELETE = データを削除
  • ステータスコード: 200=成功、400=リクエストエラー、500=サーバーエラー

5. YAML（YAML Ain't Markup Language）

YAMLの本質

YAMLとは: 設定ファイルを人間が読みやすく書く形式

database:
  host: localhost
  port: 5432
  name: mydb
  
api:
  timeout: 30
  retry: 3
  endpoints:
    - /api/users
    - /api/products

読み方のコツ:

  • インデント（字下げ）で階層を表現
  • - で配列の要素を表現
  • : でキーと値を区切る
  • JSONより読みやすい（括弧が少ない）

6. Logs（ログファイル）

Logsの本質

Logsとは: システムが「何をしたか」「何が起きたか」を記録したもの

2024-01-01 10:00:00 INFO  [main] Application started
2024-01-01 10:00:05 DEBUG [api] Request received: GET /api/users
2024-01-01 10:00:06 INFO  [db] Query executed: SELECT * FROM users
2024-01-01 10:00:07 ERROR [api] Database connection failed: timeout
2024-01-01 10:00:08 WARN  [api] Retrying connection (attempt 1/3)

読み方のコツ:

  • タイムスタンプ = いつ起きたか
  • ログレベル = 重要度（ERROR > WARN > INFO > DEBUG）
  • モジュール名 = どこで起きたか
  • メッセージ = 何が起きたか
  • ERRORから読む（問題を特定）

Day 11-14: AI技術の完全復習

1. RAG（Retrieval-Augmented Generation）

RAGの本質

RAGとは: AIに質問する際、関連情報を検索して一緒に渡す技術

質問: 「退職金の計算方法は？」
↓
1. ベクトルDBで関連文書を検索
   → 就業規則の第XX条がヒット
↓
2. 検索結果をLLMに渡す
   「以下の情報を参考に回答してください:
    就業規則第XX条: 退職金は...」
↓
3. LLMが回答を生成
   「退職金は勤続年数×基本給×係数で計算されます」

RAGのメリット:

  • 最新情報を参照できる（モデルの学習データに含まれない情報も使える）
  • ハルシネーション（嘘）を減らせる（根拠を示せる）
  • 低コスト（モデルの再学習不要）

2. プロンプトエンジニアリング

プロンプトエンジニアリングの本質

プロンプトエンジニアリングとは: AIに「何をしてほしいか」を明確に伝える技術

悪い例:
「コードを書いて」
良い例:
「Pythonで、CSVファイルを読み込んで、
年齢が30歳以上のユーザーを抽出し、
名前と年齢を表示するコードを書いてください。

入力ファイル: users.csv
カラム: name, age, email

期待する出力:
田中太郎, 35
佐藤花子, 42」

プロンプトの5つの要素:

  • 役割（Role）: 「あなたはPythonの専門家です」
  • タスク（Task）: 「CSVファイルを読み込んで」
  • 制約（Constraints）: 「年齢が30歳以上」
  • 形式（Format）: 「名前と年齢を表示」
  • 例（Examples）: 「田中太郎, 35」

3. ベクトルデータベース

ベクトルDBの本質

ベクトルDBとは: データを数値ベクトルとして保存し、意味的類似性で検索できるDB

従来のDB:
「りんご」で検索 → 「りんご」だけヒット

ベクトルDB:
「りんご」で検索 → 「りんご」「果物」「赤い」「甘い」すべてヒット

ベクトルDBの用途:

  • RAG（検索拡張生成）
  • 推薦システム（類似商品推薦）
  • 画像検索（類似画像検索）
  • 異常検知（正常データとの距離を計算）

4. Fine-tuning vs RAG

技術選定の本質

Fine-tuning: モデル自体を変更する = 知識を「記憶」させる

RAG: モデルは変更せず、情報を「参照」させる

観点 Fine-tuning RAG
コスト 高（数万円〜数百万円） 低（APIコストのみ）
更新 困難（再学習が必要） 簡単（DBを更新するだけ）
精度 高（ドメイン特化） 中（検索精度に依存）
速度 速い（検索不要） 遅い（検索 + 生成）
透明性 低（ブラックボックス） 高（参照元を追跡可能）

選定基準:

  • データ更新頻度が高い: RAG推奨
  • データ更新頻度が低い: Fine-tuning検討可能
  • 透明性が必要: RAG推奨
  • 高速応答が必要: Fine-tuning推奨

Week 2の実践演習: 統合課題

課題1: コード読解の総合問題

シナリオ

顧客から「ユーザー登録APIのエラーログを見てほしい」と依頼されました。以下のログを読んで、問題を特定してください。

2024-01-01 10:00:00 INFO  [api] POST /api/users received
2024-01-01 10:00:00 DEBUG [api] Request body: {"name":"田中太郎","email":"tanaka@example.com"}
2024-01-01 10:00:01 INFO  [db] Connecting to database...
2024-01-01 10:00:02 ERROR [db] Connection failed: FATAL: password authentication failed for user "admin"
2024-01-01 10:00:02 ERROR [api] Database error: connection refused
2024-01-01 10:00:02 INFO  [api] Response: 500 Internal Server Error

質問

  1. 何が起きたか？
  2. 原因は何か？
  3. どう対処すべきか？

解答例

1. 何が起きたか: ユーザー登録APIがデータベース接続に失敗し、500エラーを返した

2. 原因: データベースのパスワード認証が失敗している（"password authentication failed for user 'admin'"）

3. 対処方法:

  • データベースの接続設定（YAML/環境変数）を確認
  • パスワードが正しいか確認
  • データベースユーザー"admin"が存在するか確認
  • ネットワーク接続を確認（ファイアウォール、セキュリティグループ）

課題2: AI技術の選定問題

シナリオ

製造業の顧客から「製品マニュアルを検索できるAIチャットボットを作りたい」と相談されました。以下の要件を満たす技術を提案してください。

要件

  • 製品マニュアルは毎月更新される（新製品追加、仕様変更）
  • マニュアルは日本語で書かれている
  • ユーザーは「エラーコード123の対処法は？」のような質問をする
  • 回答の根拠（どのマニュアルを参照したか）を示したい
  • 予算は月額10万円以内

解答例

提案技術: RAG（検索拡張生成）

理由:

  • 更新頻度: 毎月更新 → RAGが最適（Fine-tuningは再学習コストが高い）
  • 透明性: 根拠を示したい → RAGは参照元を追跡可能
  • コスト: 月額10万円以内 → RAGはAPIコストのみで実現可能

システム構成:

  1. ベクトルDB: Pinecone（無料枠 or 月額$70〜）
  2. Embedding: OpenAI text-embedding-3-small（月額$1,000〜）
  3. LLM: GPT-4o-mini（月額$5,000〜）
  4. フロントエンド: Streamlit（無料）

月額コスト試算:

  • Pinecone: $70
  • OpenAI API: $6,000（Embedding + LLM）
  • 合計: 約$6,070（約90万円）→ 予算オーバー

コスト削減案:

  • Pineconeの代わりにChroma（無料、セルフホスト）
  • GPT-4o-miniの代わりにGPT-3.5-turbo（コスト1/10）
  • → 月額約$1,000（約15万円）→ まだ予算オーバー

最終提案:

  • Phase 1: PoC（Chroma + GPT-3.5-turbo）で効果検証（月額$500）
  • Phase 2: 効果が確認できたら予算を増額してもらう
